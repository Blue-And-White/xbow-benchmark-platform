# xbow CTF 解题平台（xbow-bench-platform）实现方案

## 目标
一个独立、多用户、带完整 UI 的 CTF 平台，托管 104 个 xbow 靶机。核心能力：注册码注册 → 登录 → 创建多个"解题情况表"(每表独立 api-key + 独立 104 题进度) → 启动靶机(注入随机动态 flag、计时) → 提交 flag 校验 → 通过则关闭容器并记录用时 / 手动关闭则清空本次 flag 与计时。平台自带反代对外访问，也支持直连端口，管理员可配置并发上限等。本地开发调试（OrbStack docker，overlay2 构建快），做好推到远程。

## 技术栈（用真框架、高性能、不手搓）
- **后端**：FastAPI（async）+ SQLAlchemy 2.0（async）+ SQLite(WAL)（自包含单容器；如需可平滑切 Postgres）。
- **前端**：Jinja2 + **HTMX + Tailwind(CDN)** —— 真 framework、无 JS 构建管线、完整交互 UI（看板/排行榜/历史/管理）。API-first，将来要换 React SPA 也不动后端。
- **容器编排**：每次启动用 `docker compose -p xben_<sheet>_<ch>_<attempt>` + 临时目录 + 生成的 override 跑靶机（天然支持同题多用户并发隔离）。
- **平台自身**：Dockerfile + compose 容器化，部署时挂 docker.sock + xbow 仓库 + flag manifest。

## 数据模型
- `users`：username, password_hash, is_admin, created_at。
- `settings`(单例)：registration_code(默认 ROOTLAB@123)、max_concurrent_per_user(默认 3)、public_base_url、allow_direct_port(bool)。
- `solve_sheets`：user_id, name, api_key(unique), created_at。**api-key 与 sheet 1:1**，该 key 下所有动作绑定此 sheet。
- `challenges`(从 104 个 benchmark.json 灌入)：benchmark_name, title, level, tags, win_condition。
- `attempts`：sheet_id, challenge_id, status(not_started/in_progress/solved/abandoned), dynamic_flag, compose_project, host_port, started_at, solved_at, solve_duration_ms。
- 每表每题状态派生：有 solved 尝试→已解；否则有 in_progress→作答中；否则未开始。
- `flag_manifest`(由扫描 104 个 Dockerfile 生成)：benchmark_name, service, flag_type(file|env|embedded), flag_path, original_flag(确定性 FLAG{sha256(name)})。
- 启动时 seed 一个管理员账号（env: ADMIN_USER/ADMIN_PASSWORD）。

## API（除 register/login/admin 外，均需 `X-API-Key`=某 sheet 的 key）
- `POST /auth/register`{registration_code,username,password} → 建账号（校验注册码）。
- `POST /auth/login`{username,password} → 会话 cookie（给 UI 用）。
- `GET/POST /sheets` → 列出/创建解题情况表 → 返回该表 api_key。
- `GET /challenges` → 本表 104 题及状态(未开始/作答中/已解) + level/tags。
- `POST /challenges/{id}/start` → 校验该用户并发未超 max_concurrent_per_user；注入随机动态 flag 起容器；status=in_progress；返回 {attempt_id, url(反代) [, direct_url]}。
- `POST /challenges/{id}/submit`{flag} → 比对本 attempt.dynamic_flag；一致→status=solved、记 solve_duration、停删容器；不一致→"incorrect"，attempt 保留。
- `POST /challenges/{id}/stop` → 手动关闭：停删容器、清空 dynamic_flag+started_at、status=abandoned（表上回到未开始）。
- `GET /leaderboard` → 按已解数+总用时排名。
- `GET/PUT /admin/settings`、`GET /admin/users` → 管理员配置 + 用户管理。

## 动态 flag 注入（不重新构建镜像，秒级）
预构建镜像用确定性 flag（common.mk 默认）。启动时生成随机 `FLAG{uuid}`，按 manifest 类型注入：
- **file 类**（`> /flag` 等）：override 里 `volumes: - <生成的flag文件>:<flag_path>:ro` 挂载覆盖。
- **env 类**（`ENV FLAG`）：override 里 `environment: FLAG=<random>`。
- **embedded 类**（`sed s/FLAG/.../g 文件`）：up 后 `docker compose exec -T <svc> sed -i "s|<original_flag>|<random>|g" <flag_path>"`（平台知道 original_flag=确定性值）。
- 所有服务恒加 `GOMAXPROCS=8`（通杀 384 核老 Go mysql panic）；CA/国内源已在构建期 baked。
- 端口：`docker compose port <svc> 80` 取宿主端口；反代 URL=`{public_base_url}/c/{attempt_id}/`；allow_direct_port 开时再返回直连 `http://host:port`。

## 反向代理
平台单端口对外：`/c/{attempt_id}/` → `http://127.0.0.1:<容器端口>/`（HTTP 反代 + 路径改写）。靶机端口默认不直暴露，除非管理员开 allow_direct_port。

## 前端（完整 UI，HTMX+Tailwind）
- 登录页、注册页（填注册码）。
- 用户主页：解题情况表列表 + 新建表（显示各表 api-key，可复制）。
- 表看板：104 题网格，每题状态色块 + 用时；点启动→作答中(轮询刷新)→内嵌提交框/停止按钮。
- 排行榜、解题历史、管理员设置页（并发上限/注册码/public_base_url/直连开关/用户管理）。

## 本地开发（/Users/qinghua/code/xbow-benchmark，OrbStack overlay2）
- **P0a**：gh-proxy 拉 xbow 仓库到本地；`scripts/gen_manifest.py` 扫 104 个 Dockerfile 生成 flag_manifest.json；本地 build 1-2 个靶机（overlay2 快；本地 apt 慢才打 CA/国内源补丁，自动探测）。
- **P0b**：搭 FastAPI 骨架 + DB + manifest 加载。
- **P1**：auth(register/login/sheet api-key)、challenges(start/submit/stop)、动态 flag 编排、反代。本地用 1-2 个靶机实测：错 flag→incorrect、对 flag→solved+清容器、手动停→清 flag+计时。
- **P2**：完整 UI。
- **P3**：平台自身容器化，本地自测。

## 部署到远程（HAI 容器）
- rsync 代码到 /root/xbow-bench-platform；平台容器挂 /var/run/docker.sock + /root/validation-benchmarks(仓库) + flag_manifest；对外一个端口跑反代。
- **预构建 30 个**（前 30，可配置）：`scripts/prebuild.py` 自动打补丁(CA/国内源/GOMAXPROCS) + `make build`，后台跑。先探 `/dev/fuse` 能否上 fuse-overlayfs（能则远程构建提速到秒级，否则接受 vfs ~5min/题后台跑）。
- 加 dockerd + 平台的开机自启（HAI 无 systemd，用 supervisor/cron @reboot）。
